hadoop

Posted on 2018-06-25

hadoop是大数据技术的基础，在大数据体系中的地位至关重要。基于个人对hadoop的学习及掌握程度，本人从hadoop 1.x开始，到2.x结束。不足之处还望路过的各位大神多多指教。

Hadoop 1.x

在hadoop1.x的版本中，hadoop主要分为hdfs和mr两部分，接下来分别就hdfs和mr分别介绍。

HDFS

HDFS全称为Hadoop Distributed File System，即Hadoop分布式文件系统，提供了一个高度容错性和高吞吐量的海量数据存储解决方案。作为一个为Hadoop提供数据存储服务的基础数据存储平台，HDFS具有具有高度容错、高吞吐量、容易扩展、高可靠性等特征，为大型数据集的处理提供了一个强有力的工具。

架构

HDFS采用主从架构体系，由NameNode、SecondaryNameNode和DataNode三个组件组成。一个HDFS集群包含一个NN和多个DN。

Namenode

NameNode是HDFS集群的master，关于NameNode需要明白的主要有以下几点：

  1. 管理着文件系统命名空间

    它维护着文件系统树及树中的所有文件和目录

  2. 存储元数据，它保存的元数据的种类有以下三种:

    文件目录名及他们之间的层级关系
文件目录的所有者及其权限
每个文件块的名及文件由哪些块组成

  3. 元数据保存在内存中

    元数据信息不包含每个块的位置信息
  4. 保存文件、block、datanode之间的映射关系

  5. Hadoop更倾向存储大文件的原因呢：

    一般来说，一条元数据会占用200byte内存空间。
假设block大小为64MB，备份数量为3，那么1GB大小的文件将占用16*3=48个block。
如果现在有1000个1MB的文件，则占用1000*3=3000个block（多个文件不能共享一个block）。
所以文件越小，存储同等大小文件所需要的元数据就越多，因block溢出而浪费的空间也越大。
  6. 元数据持久化
    在NameNode中存放元数据的文件是fsimage。
在系统运行期间所有对元数据的操作都保存在内存中并被持久化到另一个文件edits中。
edits文件和fsimage文件会被SecondaryNameNode周期性的合并
  7. 运行NameNode会占用大量内存和I/O资源
    一般NameNode不会存储用户数据或执行MapReduce任务

  8. 全Hadoop系统只有一个NameNode

    容易导致单点问题，单点问题的解决方案有如下：
（1）将Hadoop元数据写入到本地文件系统的同时再实时同步到一个远程挂载的网络文件系统（FS）
（2）运行一个SecondaryNameNode，负责与NameNode交互，定期通过编辑日志文件合并命名空间镜像，
当NameNode发生故障是它会通过自己合并的命名空间镜像副本来回复。
需注意的是SecondaryNameNode保存的状态总是滞后于NameNode，所以这种方式难免会导致丢失部分数据。
